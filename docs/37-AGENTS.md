## 3. Core Development Rules

1.  **Enforce Clean Architecture:** Maintain strict boundary separation:
    $$\text{Presentation Layer (UI, Notifiers)} \longrightarrow \text{Domain Layer (Use Cases, Entities)} \longleftarrow \text{Data Layer (Repositories, DTOs)}$$
2.  **Isolate Domain Imports:** The Domain layer must remain pure Dart. The AI agent must **never** import Flutter UI packages (such as `package:flutter/material.dart` or `package:flutter/widgets.dart`) into domain files.
3.  **No Direct API Calls:** Widgets must never call remote APIs or data sources directly. All network requests must pass through repository layers.
4.  **No Hardcoded Secrets:** Do not hardcode API keys, service role keys, or database credentials. Use backend Edge Functions to proxy API requests securely.
5.  **Secure Local Caching:** Sensitive credentials (like user JWT session tokens) must be saved in secure storage interfaces (like `flutter_secure_storage`).

---

## 4. Incremental Code Generation Rules
*   **Focused Chunks:** Avoid overwriting whole files when making edits. Modify only target code blocks to minimize code churn.
*   **Single Feature Focus:** Complete one feature at a time. Ensure a feature compiles and passes tests before beginning the next task.
*   **Zero Duplicate Code:** Before writing new helper functions, search `core/utils/` and `shared/` to reuse existing components.

---

## 5. Coding Output Expectations

Every feature code generation request must produce a complete set of components:
*   [ ] **Presentation Page:** Screen layout widgets using centralized theme tokens.
*   [ ] **State Controller:** Riverpod AsyncNotifier managing screen states.
*   [ ] **Domain Entity:** Pure Dart class representing the feature's data.
*   [ ] **Use Case:** Executable business logic block (e.g., `SendChatMessage`).
*   [ ] **Repository Interface:** Abstract interface class.
*   [ ] **Repository Implementation:** Concrete class resolving remote APIs and local storage caches.
*   [ ] **Remote Data Source:** API calls using Dio clients.
*   [ ] **Data Transfer Object (DTO):** Serialized model using Freezed.
*   [ ] **Unit Tests:** Business logic verification tests meeting target code coverage (>90%).

---

## 6. Security, RLS & Secrets Verification Checklist

The AI agent must verify database scripts against this checklist before proposing migrations:
*   [ ] Is Row-Level Security (RLS) enabled on all user-owned tables?
*   [ ] Do policies restrict reads and updates to the authenticated user (`auth.uid() = user_id`)?
*   [ ] Do migration scripts include matching rollback files?
*   [ ] Are PII data and user transcripts excluded from application logs?

---

## 7. Performance & Caching Rules
*   Use `const` constructors on all stateless widgets to minimize rebuilds.
*   Lazy-load and paginate large lists (such as chat history logs).
*   Save learning progress (lessons, SRS vocabulary queues) in offline databases (Hive/Isar) and sync automatically when online.

---

## 8. Quality Checklist & Definition of Done

Before completing a task, the AI agent must verify that:
*   ✅ Code compiles successfully without warnings or analyzer errors.
*   ✅ Linter rules pass according to `analysis_options.yaml`.
*   ✅ Automated unit, widget, and integration tests pass.
*   ✅ The UI is responsive across different screen sizes.
*   ✅ No API keys or secrets are exposed in git commits.
*   ✅ Documentation, comments, and changelogs have been updated.
